# /docs — Universal Project Documentation Decks

Generate navigable Reveal.js documentation decks from project artifacts. One deck per numbered folder, one hub deck as master navigation, honest change tracking across everything. Works with any project — auto-detects folder structure and content.

> CSS template (slide decks): `~/projects/rapid-workflow/templates/template-docs-deck.html`
> CSS template (web doc site): `~/projects/rapid-workflow/templates/template-docs-page.html` — the **standard** spec-style layout (top forge-nav + left sidebar menu + main) for every project's documentation **web pages**. Convert existing pages in place with `tools/apply-docs-sidebar.py <file> --title <T>`.
> Harness identity & product links: the doc site is the **developer view** (navigate + develop the product). Every page includes `<script src="env-links.js" defer></script>` (`docs/env-links.js`), which tags the harness "Developer View" and renders a **Product** cluster linking to the *finished product* per environment — Local / Dev / Production — from `docs/env.json` (`{local:{url,launch}, dev:{url}, prod:{url}}`). Local exposes a copyable launch command. A built application project fills `env.json` from its deploy targets (local dev server + dev/prod deployments). `env-links.js` also adds a per-page **↻ Regenerate** button that regenerates *that* page from its source via `observe-server` `POST /api/regen?page=<file>`, running the page's registered command from `docs/regen.json` (deterministic script · page-specific `claude -p` · or manual). **Any deterministic generator must emit the full harness** (forge-nav + sidebar + `env-links.js`) or regenerating the page strips it.
> Document roles (standard separation): **PRD** = original requirements · **Enhanced PRD** = requirements organized as a Business→Functional→Technical pyramid (priority as a badge) **with change tracking** — a per-requirement change history + an issue-aligned Change Log (every change names its GitHub issue and the BR/FR/TR it updated) · **Documentation** = the full, organized description of the *finished product* (product-level, **no change logs**) · **Spec** = implementation. Requirement changes flow from the gap loop (`.forge/GAPS.json` → `gaps-to-issues.sh` → GitHub) into the Enhanced PRD Change Log.
> System reference: `~/projects/rapid-workflow/docs/forge-architecture.html` (D17, D18)
> Shares CSS vocabulary with `/decision` decks (decisions + panels).

## Invocation

```
/docs build                  — generate all per-folder decks + hub from project artifacts
/docs build <folder>         — generate deck for a specific folder only
/docs hub                    — regenerate hub deck only
/docs changelog              — show master CHANGES.md
/docs status                 — print folder inventory with completeness
```

## First Actions on Invocation

1. **Find the project root.** Walk up from cwd looking for `CONSTITUTION.md`, `.forge/`, or numbered folders (`00-*/`, `01-*/`, etc.). Fall back to cwd.
2. **Scan for numbered folders.** List all directories matching `NN-*` pattern (00 through 99). Also check for `decisions/`, `panels/`, `tests/`.
3. **Read project name.** From `00-vision/VISION.md` first heading, or `CLAUDE.md` first heading, or directory name.
4. **Read pillars.** From `00-vision/PILLARS.md` if it exists. These are used as tags throughout all decks.

## Universal Slide Generation Rules

These rules apply to ALL generated decks. Follow them exactly.

### CSS

Copy the complete `<style>` block from `~/projects/workflow/templates/template-docs-deck.html` into every generated deck. This is the single source of truth for all visual styling. Do not invent CSS classes — use only what the template defines.

### Reveal.js

Every deck ends with:
```html
<script src="https://cdn.jsdelivr.net/npm/reveal.js@5.0.4/dist/reveal.min.js"></script>
<script>Reveal.initialize({hash:true,width:1280,height:720,margin:0,minScale:0.2,maxScale:2,center:false,transition:'none',controls:true,controlsTutorial:false,progress:true});</script>
```

### Slide Structure

Every slide is a `<section><div class="slide">` containing:
1. Phase breadcrumb (`.phase-nav`) — shows all numbered folders, current one `active`
2. Eyebrow (`.eyebrow`) — folder name + slide type
3. Title (`.title`) — action title (what the slide says, not what it shows)
4. Content area — uses template CSS classes
5. Footer (`.footer`) — project name + folder/slide ID

### Phase Breadcrumb

Render from the actual numbered folders found in the project. Each folder becomes a `.ph` element. The current deck's folder gets `.active`. Use short labels derived from folder names (e.g., `00-vision` → `Vision`, `04-spec` → `Spec`).

### Action Titles (McKinsey rule)

Every slide title is a **complete sentence stating the slide's conclusion**, not a topic label. "Requirements table" is wrong. "Three MUST constraints eliminate 90% of architecture options" is right. Extract the action title from the content — what is the single most important thing this slide says?

### Plain-English-First Pattern

Every section and slide must lead with a **plain-English explanation** before any technical detail. Write for someone who walked into the room cold — what is this and why should they care? Then follow with "**Technically:**" and the precise description. This applies to slide bodies, section intros, and all generated documentation. Reference: `~/projects/workflow/docs/forge-architecture.html` (`.plain-english` + `.technical-detail` CSS classes).

### Diagrams

Generate inline SVG for architecture, workflow, and dependency diagrams. Use the template's color variables (`var(--navy)`, `var(--teal)`, `var(--accent)`, etc.) for consistency. The `.diagram-frame` class provides the container. Keep SVGs under 200 lines — diagram clarity matters more than detail.

### Content Extraction

Read the actual markdown files in each folder. Parse headings, lists, tables, YAML frontmatter. Extract the content — don't summarize or paraphrase. The deck is a visual rendering of the artifacts, not a separate interpretation.

## Per-Folder Deck Specifications

### `00-vision/deck.html` — Vision & Pillars

**Slides:**
1. **North Star** — The project objective as a single prominent statement (use `.metric` class for the big statement). Below: success signals as `.cascade-item` entries.
2. **Pillars** — One `.option.navy` card per pillar in a `.options` grid. Each card: pillar name as `.opt-name`, description as `.opt-desc`, "why this matters" as a `.pro-con .item.pro` entry.
3. **Values** (if `VALUES.md` exists) — Same card layout with `.option.teal`.

**Content source:** `VISION.md`, `PILLARS.md`, `VALUES.md`

---

### `01-intake/deck.html` — PRD & Requirements

**Slides:**
1. **Problem Statement** — Extract the problem statement paragraph. Render as `.rationale` callout. Below: target user as a `.cascade-item`.
2. **Requirements** — `.decision-index` table with columns: ID, Requirement, Priority (MUST/SHOULD/MAY), Pillar, Status. Color the priority cell: red for MUST, amber for SHOULD, gray for MAY.
3. **Constraints** — One `.option` card per constraint in a `.options.cols-2` or `.cols-3` grid. Constraints that eliminate architecture options get `.option.block` styling. These are the most important cards in the deck.
4. **Out of Scope** — Bulleted list in a `.rationale.accent` callout. Explicit exclusions save weeks.
5. **Success Criteria** — `.cascade` grid with one `.cascade-item.green` per measurable criterion.

**Content source:** `PRD.md`, `STAKEHOLDERS.md`
**Badge:** `.badge.locked` (PRD is frozen)

---

### `02-grounding/deck.html` — Research & Evidence

**Slides (one per research question):**
1. **Question** — The question as the title. Classification badge: `.badge.decided` for VERIFIED, `.badge.open` for UNVERIFIED, `.badge.blocked` for OPEN.
2. **Options Matrix** — `.options.cols-3` grid with one `.option` per option. Chosen option gets `.option.chosen`. Pro/con lists show cost, performance, complexity trade-offs.
3. **Evidence** — `.rationale` callout with sources, version-pinned. `.r-source` for each citation.

**Content source:** `02-grounding/*.md` files

---

### `03-panels/deck.html` — Panel Synthesis Overview

This deck summarizes the panel synthesis. Individual panel findings are in `panels/deck.html` (generated by `/decision panel`).

**Slides:**
1. **Synthesis Overview** — `.metrics` row showing: panels run, experts consulted, convergent findings, divergent findings, open asks.
2. **Convergent Findings** — `.cascade` grid with one `.cascade-item.green` per finding all panels agreed on.
3. **Divergent Findings** — `.cascade` grid with one `.cascade-item.amber` per disagreement. Each includes which panels diverge and the recommended resolution.
4. **Critical Risks** — `.decision-index` table: risk, severity, pillar, owning panel, mitigation.
5. **Link slide** — Points to `panels/deck.html` for full per-expert findings.

**Content source:** `synthesis.md`, cross-reference `panels/P-*.md`

---

### `04-spec/deck.html` — Architecture, Workflow, Interfaces

This is typically the largest deck. Generate slides in this order:

**Architecture slides:**
1. **System Overview** — SVG architecture diagram in `.diagram-frame`. Show components as rounded rectangles, data flow as arrows, external services as dashed borders. Use navy for core components, teal for data stores, accent for external services.
2. **Component Map** — `.options` grid with one `.option.navy` card per component. Each: name, responsibility (1 sentence), owned files, interfaces exposed.
3. **Data Model** — `.decision-index` table showing entities, relationships, storage.

**Workflow slides:**
4. **User Journey** — SVG state machine diagram. Nodes as `.sm-node` rectangles, edges as arrows. Golden path in navy, error paths in red, recovery paths in teal.
5. **Per-Node Detail** (one slide per key workflow node) — `.split` layout. Left: the node with in/proc/out as `.cascade-item` entries. Right: edge cases and failure paths as `.pro-con` items.

**Interface slides:**
6. **Contracts** — `.decision-index` table: endpoint/function, input type, output type, error contract, owning component.
7. **Spec Traceability** — `.decision-index` table: spec section, PRD requirement, status, implementing task. Use `.req-trace` spans for `[FROM PRD]`, `[DERIVED]`, `[OPEN]` tags.

**Content source:** `spec.md`, `architecture.md`, `workflow.md`, `CONTRACTS.md`, `INTERFACES.md`, `DATA_MODEL.md`, `REQUIREMENTS.md`
**Badge:** `.badge.stable` or `.badge.draft` based on whether DIFF.md has unresolved entries

---

### `05-build/deck.html` — Build Plan & Progress

**Slides:**
1. **Task Overview** — `.metrics` row: total tasks, completed, in progress, blocked. Below: `.decision-index` table of all tasks with status, agent assignment, spec reference.
2. **Dependency Graph** — SVG diagram showing task dependencies. Completed tasks in green, in-progress in navy, blocked in red, pending in gray.
3. **Agent Map** — `.options` grid with one card per agent. Each: agent name, assigned tasks, owned files, status.
4. **Decisions Log** — Link to `decisions/deck.html`. Show count of decided vs open decisions.
5. **Build Timeline** — `.cascade` grid showing phase progression with timestamps from `.forge/MEMORY.md`.

**Content source:** `.forge/TASKS.json`, `.forge/MEMORY.md`, `.forge/STATE.json`, `PLAN.md`, `DECISIONS.md`, `RISKS.md`

---

### `06-walkthrough/deck.html` — Scenarios & Validation

**Slides (one per scenario):**
1. **Scenario Card** — `.split` layout. Left: persona card (`.option.teal` with name, role, goal, starting state). Right: click-by-click steps as numbered `.cascade-item` entries with inner-voice text in italic.
2. **Result** — Pass/fail badge per step. Findings as `.pro-con` items (`.item.pro` for pass, `.item.con` for fail with the specific issue).

**Summary slide:**
3. **Walkthrough Results** — `.metrics` row: scenarios run, steps passed, steps failed, gaps found. Below: `.decision-index` table of all findings with severity and gap reference.

**Content source:** `scenarios.md`, `agent-runs/*.md`, `human-pass.md`, `EVAL.md`

---

### `07-gaps/deck.html` — Gap Loop Status

**Slides:**
1. **Gap Overview** — `.metrics` row: total gaps, resolved, open, by severity (BLOCKER/HIGH/MEDIUM/LOW). Below: pillar breakdown showing which pillars have the most gaps.
2. **Per-Gap Cards** (one slide per BLOCKER/HIGH gap, grouped onto slides for MEDIUM/LOW) — `.option` card per gap. Card class by severity: `.option.block` for BLOCKER, `.option.caution` for HIGH, plain for MEDIUM/LOW. Each card: gap title, classification (spec-level vs PRD-level), pillar tag, resolution action.
3. **Resolution History** — `.changelog` entries showing gap loop iterations. What was re-derived, what was routed to stakeholder.
4. **Remaining Asks** — `.decision-index` table of unresolved items for Gate 3.

**Content source:** `GAPS.md`, `HISTORY.md`, `RETROSPECTIVE.md`, `gaps/*.md`, `.forge/GAPS.json`

---

### `tests/deck.html` — Test Coverage & Results

**Slides:**
1. **Coverage Matrix** — `.decision-index` table: spec section, test file, test count, status (pass/fail/missing). Green dot for passing, red for failing, gray for missing. This is the traceability layer — every spec section must map to at least one test.
2. **Results Dashboard** — `.metrics` row: total tests, passing, failing, coverage %. Below: failing tests as `.option.block` cards with the failure message.
3. **Visual QA** (if applicable) — `.options.cols-3` grid with screenshots at mobile/tablet/desktop viewports. Pass/fail badge per viewport.
4. **Eval Harness Status** — `.rationale` callout showing whether the eval harness is locked (immutable) and last run timestamp.

**Content source:** `.forge/EVAL/`, `.forge/TEST_RESULTS.md`, test output, screenshot evidence

## Hub Deck (`docs/hub.html`)

The master navigation deck. Entry point for the entire documentation corpus.

**Slide 1 — Project Overview:**
- Eyebrow: `Documentation Hub`
- Title: `{PROJECT NAME} — {one-line objective from VISION.md}`
- `.metrics` row: folders, total files, decisions, panel runs, gaps, test coverage %
- `.rationale` callout: current forge phase + status from `.forge/STATE.json`

**Slide 2 — Folder Navigation:**
- `.hub-grid` with one `.hub-card` per numbered folder + `decisions/` + `panels/` + `tests/`
- Each card shows: folder name, file count, key file names, status badge, progress bar
- Cards link to the folder's deck.html (via Reveal.js hash navigation or note in the card)
- Empty folders get `.hub-card.empty`. Complete folders get `.hub-card.complete`. Current phase gets `.hub-card.active`.

**Slide 3 — Cross-Reference Map:**
- SVG diagram showing how folders reference each other. PRD → Panels → Spec → Build → Gaps → (loop back). Decisions and panels cross-link to all phases.

**Slide 4 — Changelog Digest:**
- `.changelog` entries showing the last 20 entries from `CHANGES.md`
- Each entry: date, phase, what changed

**Slide 5 — Deck Index:**
- `.decision-index` table listing every generated deck, its file count, last updated date, and slide count.

## Change Tracking

### `CHANGES.md` — Master Changelog

Append-only file at project root. Every meaningful artifact change gets an entry.

**Format:**
```
## CHANGES

| Date | Phase | File | Change |
|------|-------|------|--------|
| 2026-05-26T14:30Z | P4 Spec | 04-spec/architecture.md | Derived from PRD + workflow. 3 components defined. |
| 2026-05-26T15:10Z | P6 Build | src/camera/pipeline.ts | Implemented camera pipeline per SPEC §4.2 |
| 2026-05-26T16:00Z | P8 Gaps | 07-gaps/GAPS.md | Gap G-03 classified spec-level, re-deriving §3.4 |
```

**Two sources feed CHANGES.md:**
1. **Agents** — every forge phase appends what it produced and why. This is the semantic layer.
2. **Hook** — catches every file write to numbered folders and appends a mechanical entry (timestamp + path). This is the honesty layer — even if an agent forgets, the hook catches it.

### Per-Folder Changelogs

Each numbered folder gets a `CHANGELOG.md` maintained by forge agents as part of their phase output. Not auto-generated by hooks — agents write meaningful descriptions of what changed and why.

### Deck Changelogs

Every generated deck includes a final "Changelog" slide showing the last 10 entries from that folder's `CHANGELOG.md`.

## Hook Configuration

To enable automatic change tracking and deck regeneration, configure a hook in the project's `.claude/settings.json` or the user's global settings:

**After Write/Edit hook** — when any file in a numbered folder, `decisions/`, `panels/`, or `tests/` changes:
1. Append a timestamped entry to `CHANGES.md` (file path + timestamp)
2. Regenerate `docs/hub.html` (the hub only — per-folder decks are regenerated by agents or via `/docs build`)

The hook captures the mechanical "what file changed when" record. Agents provide the semantic "what changed and why" in per-folder CHANGELOGs and in CHANGES.md entries with phase attribution.

## Universality

This skill works with **any project**, not just forge builds:

- **Partial structures:** If a project only has `00-vision/` and `01-intake/`, only those decks are generated. The hub shows what exists.
- **Non-forge projects:** If there's no `.forge/` directory, the skill still works. It reads whatever numbered folders exist and generates decks from their markdown content.
- **Custom folders:** Any `NN-*` directory is picked up automatically. The phase breadcrumb renders whatever folders exist.
- **No hardcoded content:** The skill reads project artifacts and renders them. It never invents content that isn't in the source files.
- **Idempotent:** Running `/docs build` twice produces the same output. Decks are regenerated from source files, not incrementally patched.

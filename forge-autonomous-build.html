Three arcs. P0–P3 = Understand (challenge requirements). P4–P5 = Specify (derive buildable contract). P6–P9 = Execute + Converge (build, test, loop, ship). The gap loop (P8) re-derives the spec on failure — the code follows. Arc 3 is cyclical; Arcs 1–2 are linear.
Eval-first / TDD invariant. Before P6 begins, P5 generates an immutable eval harness from the workflow state machine (P4). Tests are written before code, generated from spec — not by the implementing agent. Agents cannot modify their own success criteria. The branch only advances on verified improvement (keep-or-revert ratchet). Sources: Karpathy autoresearch loop (program.md: "prepare.py is read-only"); Kent Beck, Test-Driven Development (2002): red-green-refactor with the test surface owned by a separate concern.
Inter-stage assertions. Between each phase, programmatic contract checks run automatically. After P4: every PRD requirement maps to a spec section or explicit out-of-scope tag. After P5: every task references a spec section; dependency graph has no cycles. After P6: every file in the repo is owned by a spec section. Assertion failure = automatic retry with the violation message appended. Source: Zaharia et al., "The Shift from Models to Compound AI Systems" (Berkeley AI Research, 2024); DSPy Assert/Suggest pattern.

Research agent output format: Question (from panel), Options (min 3 with cost/perf/complexity matrix), Evidence (primary sources, version-pinned), Recommendation (with falsifiable assumptions), Fallback (what to do when recommendation fails), Verification (how to confirm at runtime).

D7b — Test-Driven Development
Tests generated from workflow state machine before any code is written
References D1 P4–P5. The eval harness is immutable once generated — implementing agents cannot weaken, skip, or modify tests. This is the TDD red-green-refactor cycle, enforced structurally rather than by discipline. Sources: Kent Beck, TDD (2002); Karpathy autoresearch "prepare.py is read-only."

The trust contract. At each gate, the operator receives evidence, not claims. G1: "Here are the findings with sources. Here's where panels disagree. Here's what couldn't be verified." G2: "Here's the spec with every requirement traced. Here are the tests. Here's the cost." G3: "Here's the running app. Here's the audit. Here are the gaps. Here's actual spend." Constitution Article VIII (Honest Reporting) enforces this: agents describe what actually happened, not what was intended.

D17 — Panel Documentation Corpus
Every expert panel invocation produces a persistent, navigable record — organized by expert and by topic
References D1 P2, D6. When any expert panel skill runs (technical, business, SME, user, or project-specific), it writes a structured record to panels/. A navigation index auto-rebuilds on every addition. Two views: by-expert (what did Marcus say across all reviews?) and by-topic (all experts on the camera pipeline).

D17b: Panel record structure (one file per invocation)

YAML Frontmatter

id: P-01 panel_type: technical-expert-panel topic: "Camera pipeline architecture" phase: 2 date: 2026-05-26 pillars: [Build Quality, Correctness] experts: - name: Marcus lens: iPad/Mobile verdict: caution finding: "Canvas touch events drop frames…" asks: ["Use CSS transforms", "Test real iPad"] - name: Alex lens: SWE Skeptic verdict: pass finding: "" asks: [] convergent: ["Touch targets must be ≥44px"] divergent: ["Canvas vs CSS transforms"] recommended_action: "Prototype both…"

Navigation Structure

Cover slide: Index table of all panel runs (topic, type, date, expert count, open asks)

By-topic slides: One slide per panel invocation. All experts for that topic, with verdicts color-coded (green = approve, amber = caution, red = block, gray = pass).

By-expert slides: One slide per named expert across all invocations. Shows every finding and ask from that expert, grouped by topic. Answers: “What has Marcus flagged across this entire build?”

Asks tracker: Final slide aggregating all open asks from all panels, sorted by severity. Links back to the originating panel slide.
D17c: Expert verdict classification
APPROVE
No objection on this topic.
CAUTION
Flag raised. Has asks.
BLOCK
Stops the build. Requires resolution.
PASS
Nothing useful to add.
Integration with D5 (Decision Router): When a panel finding triggers an architectural or strategic decision, the decision is logged via /decision log (D5) with a back-reference to the panel record. The panel corpus and decision deck are separate artifacts but cross-linked: panel records cite the decisions they spawned; decision slides cite the panel findings that informed them.
Why per-expert views matter: In a multi-panel build, the same named expert (e.g., Marcus the iPad engineer) speaks on 5–10 different topics. The by-expert view surfaces patterns: “Marcus flagged touch targets in 4 of 6 reviews — this is a systemic issue, not a one-off.” Without this view, systemic concerns are scattered across individual panel records and invisible.
Skill: /decision panel writes the record and regenerates the deck. Invoked automatically by every panel skill after producing findings. Manual invocation for ad-hoc panels or external expert input.

Every deck’s final slide shows that folder’s last 10 changelog entries.
McKinsey action titles: Every slide title is a complete sentence stating the slide’s conclusion, not a topic label. “Architecture” is wrong. “Three components with strict contract boundaries eliminate cross-agent file conflicts” is right. The title is the takeaway; the content is the evidence.
Universality: The /docs skill works with any project. It scans for numbered folders, reads whatever markdown is there, and generates decks. No hardcoded project references, no assumptions about folder names beyond the NN-* pattern. Partial structures produce partial documentation — empty folders get placeholder slides.
Skill: /docs build generates all decks. /docs hub regenerates navigation only. /docs status shows folder inventory. CSS source: ~/projects/workflow/docs-deck-template.html.

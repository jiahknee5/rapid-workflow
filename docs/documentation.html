The Product
RAPID turns a product idea into a shipped, tested, documented app — with four human check-ins
RAPID (Rapid Autonomous Pipeline for Iterative Development), driven by the FORGE build skill, takes a product idea or a requirements document and builds the entire application — planning, coding, testing, and deploying — with minimal human input. The operator checks in at four decision points; everything else runs autonomously.

One-paragraph description

Give RAPID an idea or a PRD. It understands what to build by consulting expert panels and doing research, writes a plan and gets your approval, assigns the work to a team of AI agents that build in parallel, reviews every piece of code through multiple specialized lenses before merging, tests everything against criteria derived from the plan, fixes problems automatically when it can and flags them when it can't, and deploys the result. While it builds you watch everything live; documentation is written as it goes, not bolted on at the end; and tests are generated before any code is written, so "done" is defined before anyone starts.

Who it's for
An AI engineering lead who wants to turn a stakeholder's requirements into a shipped, tested, documented application using AI agents — without babysitting every step.

Founding pillars

Enforcement over prose. Every safety-critical step is mechanically enforced — a hook, a dependency, or a separate agent — so nothing load-bearing rests on a prose reminder that fails under pressure.

The writer is never the auditor. Implementation and review are structurally split, so the operator can walk away between gates because safety is structural, not dependent on vigilance.

The gap loop is the product. The build is the artifact; surfacing and closing gaps is the value. Failures become tracked, classified gaps that converge to resolution and feed learning, so each build starts smarter than the last.

Capabilities
Six core capabilities define what RAPID does
Each capability maps to a function the operator can rely on, in plain English. Together they cover the whole arc from idea to a deployable, observable, documented result.
1. Build the application
Given an idea or PRD, the system understands what needs to be built (expert panels + research), writes a detailed plan and gets human approval before starting, assigns work to a team of AI agents that build in parallel, reviews every piece of code through multiple specialized lenses before merging, tests everything against criteria generated from the plan, fixes problems automatically when possible and flags them for humans when not, then deploys the result.
2. Live dashboard while it builds
Open a dashboard and immediately see what phase the build is in, which agents are working on what, whether anything is blocked or broken, how much of the plan is complete, and a timeline of the entire build like a Gantt chart. The dashboard is prepopulated the moment the build starts — the full plan is laid out up front, not an empty screen that fills in. As the build progresses, bars fill and status updates in real time.
3. Documentation as it builds, not after
Every phase produces documentation automatically — no separate command. Docs update whenever a phase completes, so a mid-build look at the documentation reflects everything completed so far. At the end, all documentation is reorganized into a clean, navigable site that can be handed to a stakeholder.
4. Test results visible everywhere
Test pass/fail status appears in the dashboard, in the documentation, and at every decision point — not buried in a log file. If 3 of 50 tests are failing, that shows up from any view: the dashboard health screen, the phase status, and the documentation coverage page.
5. Combined gate view
Build status, documentation, test results, and the dashboard all know about each other. When a phase completes, all four update together. At a decision point the operator sees one combined view — the working app, test results, doc completeness, cost, and remaining gaps — instead of four separate things to cross-reference.
6. Cost & token visibility
Token and cost burn is tracked and shown per step, per phase/section, and per project, surfaced in the developer view so the operator always knows what the build is costing.

How It Works
A terminal-per-agent team, with the writer structurally separated from the auditor
RAPID builds with a small team of AI agents. Each persistent agent runs in its own terminal and the team is connected as one unit over the claude-peers message bus, with per-terminal status, monitoring, and continuous-build hooks keyed by role.

Why the split matters

The agents who write the code are never the ones who sign off on it. An independent watchdog agent — with no incentive to declare its own work done — is spawned automatically during the build and records its findings to an audit trail. The build cannot exit its critical phase without those audit entries present.

How It Works
Quality is guaranteed by mechanism, not by reminders
Each guarantee below is enforced structurally — through a gate, a dependency, or a separate agent — so the operator can trust the build between check-ins. These are product-level promises, not optional best practices.
G1 Phase gate — no skipping ahead Must
The build cannot advance into a phase whose required artifacts don't yet exist. The progression of prerequisites is enforced — locked PRD before research synthesis, synthesis before spec, spec before tests and tasks, and so on. A phase with missing inputs is blocked, not waved through.
G2 Independent watchdog — the writer is never the auditor Must
A separate auditing agent with no implementation incentive is spawned automatically during the build and records its audit. The build's critical phase cannot exit until that audit exists.
G3 Module conformance — every module traces back to a requirement Must
As each module completes, its references to the spec, PRD, and architecture must resolve to real anchors. Orphaned work — code that doesn't trace to a requirement — is filed as a conformance gap rather than silently accepted. Every MUST deliverable in the PRD must map to at least one task; an unmapped deliverable is a blocking gap.
G4 Stub gate — placeholder code can't ship Must
Placeholder code is detected as it's written, and the ship gate blocks release on any open stub gap in a MUST module. "Compiles" is not allowed to masquerade as "works."
G5 Continuation — the build doesn't quietly stop early Must
A synchronous stop hook prevents stopping while the current phase's artifact is still missing. Every halt is logged, and the system escalates after repeated nudges, so the build either makes progress or surfaces why it can't.
G6 Concrete walkthrough — every surface is exercised before release Must
Before the release gate, an explicit step-by-step walkthrough runs across every surface of the live app and records what was checked.

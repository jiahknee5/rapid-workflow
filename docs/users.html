Every user of RAPID/FORGE at a glance

FORGE turns a product idea or PRD into a shipped, tested, documented application using a team of AI agents, with the human checking in at four decision points. The people below are the humans in that loop. The agents that do the building are not in this table — see the system-actor note at the end of this section.

UserRoleWhat they touchWhat they get
AI Engineering Lead / Operator Primary — starts and runs the build /forge "<idea>" or a PRD; the live dashboard; the four decision points A deployed app, a documentation site, a test report, and a timeline of the build
Stakeholder / Requirements Owner Supplies requirements, receives the result The intake PRD; the final handoff documentation site A clean, navigable doc site they can read without manual cleanup
Gate Reviewer Approves or rejects at the four decision points The combined gate view: working app, tests, doc completeness, cost, remaining gaps One screen to decide go / no-go, instead of four artifacts to cross-reference
End User of the built product Downstream — uses whatever FORGE built The shipped application itself (project-specific) The working product (e.g. a math tutor) — placeholder, defined per project
System actors, not human users. The orchestrator (team lead), supervisor (task assignment + smoke tests), watchdog (R2 auditor), and the implementors (impl-1..impl-N, fan-out ≤ 4) are internal SYSTEM ACTORS. They run inside the build and coordinate over the claude-peers bus. They are not in the user list above; they appear in WF-4 only because the operator monitors them.
Users · Persona 1 (primary)

AI Engineering Lead / Operator

The primary user. Turns a stakeholder's requirements into a shipped, tested, documented application using AI agents — without babysitting every step.

Goals

Start a build from one line or a PRD, walk away, and come back to a deployed app plus a complete doc site, test report, and build timeline. Stay in control at the four decision points without micromanaging every task.

Context

Runs /forge "build a math tutor for kids" from Claude Code, with the live dashboard open in a browser. The build runs autonomously across its phases; the operator checks in at the four gates.

Needs

A prepopulated dashboard (the full plan visible from the start, not an empty screen). Real-time phase, agent, and test status. Documentation that updates as each phase completes. One combined view at each decision point.

Frustrations

Four systems — build, docs, tests, dashboard — that work independently but don't talk to each other. Test results buried in a log file. Having to cross-reference four artifacts at a gate. Docs that have to be cleaned up by hand at the end.

Users · Persona 2

Stakeholder / Requirements Owner

Owns what needs to be built and receives the finished result. The reason the build exists.

Goals

Hand over a product idea or PRD and, at the end, receive a deployed application backed by a clean, navigable documentation site they can read and trust.

Context

Supplies the intake PRD that seeds the build. Typically not present during the build itself; engages at the start (requirements) and at handoff (the doc site).

Needs

Their requirements understood and reflected in the plan. A documentation site they can navigate without help, reorganized into a clean deck at the end. Test status they can see plainly, not in a log.

Frustrations

Documentation that needs manual cleanup before it can be handed over. Not being able to tell what was actually built versus what was promised. Having to ask the operator to interpret raw logs.

Users · Persona 3

Gate Reviewer

The human who approves or rejects at the four decision points. Often the same person as the operator, but a distinct role: the deciding hat.

Goals

Make a confident go / no-go call at each decision point — especially the ship gate — without digging through four separate systems.

Context

Engages only at the four decision points. At the ship gate they expect one screen with: the working app, test results, doc completeness, cost, and the remaining gaps.

Needs

One combined gate view, not four things to cross-reference. Test pass/fail visible at the decision point itself. A clear read on remaining gaps before approving.

Frustrations

Cross-referencing the dashboard, the docs, the test output, and the cost separately. Approving without knowing what's still broken. Decision points that surface less than the dashboard already knows.

Users · Persona 4 (template placeholder)

End User of the Built Product

The downstream person who uses whatever FORGE shipped. This persona is project-specific and is a template placeholder — for the example build /forge "build a math tutor for kids", this would be a child (and their parent or teacher) using the math tutor.

Goals

Use a working, deployed product that does what the original idea promised. (Defined per project — fill in for the specific app being built.)

Context

Never touches FORGE itself. Touches only the shipped application after deploy. The build's golden assertions are ultimately about this person's experience working.

Needs

An app that behaves as specified, with the features the PRD described actually present and functioning. (Project-specific.)

Frustrations

Features that were planned but never shipped; behavior that doesn't match the promise. These should surface as failing golden assertions before the end user ever sees them.

Reminder. This persona is a placeholder. Each concrete FORGE build replaces it with the real downstream user(s) of that specific product.

Workflow · WF-2

PRD → gated build that hits a failing gate, runs the gap loop, re-converges

A build does not always pass a decision point on the first try. When a gate fails — for example, 3 of 50 tests failing — the system runs the gap loop, fixes what it can, flags what it can't, and re-converges on the gate. The gap loop is the feedback mechanism, not an error path.

Workflow · WF-4

Operator monitors a live build and handles a drift/stall alert

The operator watches a running build through the dashboard and handles a stalled terminal. Under the terminal-per-agent model, every persistent agent runs in its own tmux window inside one forge-build session, so a stall is localized to a single terminal and visible by role.

Build execution model — terminal-per-agent

Every persistent agent runs in its OWN terminal (tmux window/pane) inside one forge-build session, created at P6a and torn down at P6 end — replacing the old fixed 3-terminal setup. Agents: orchestrator (team lead), supervisor (task assignment + smoke tests, owns the implementors), watchdog (R2 auditor, reports to orchestrator), and each implementor impl-1..impl-N (fan-out ≤ 4). Each implementor uses its own git worktree for isolation; the terminal is long-lived so it can be monitored, message peers, and build continuously — not a fire-and-forget subagent.

The terminals form ONE agent team over the claude-peers MCP bus. On start each agent calls set_summary(role + task) and list_peers to discover teammates; coordination is structured send_message/check_messages (shoulder-tap protocol — answer peer messages immediately). Addressing derives from FORGE_ROLE per terminal.

Tests · From workflows to golden tests

From workflows to golden tests

Each workflow above declares an expected data-out. That expected data-out is exactly what becomes an acceptance assertion. The workflow map is the source; the test stubs in 08-tests and the immutable suite in .forge/EVAL are derived from it. The expected outcome of a workflow IS its golden assertion.

WorkflowGolden assertion (expected data-out)
WF-1 — idea → appAfter /forge "<idea>" and four approvals, the operator has a deployed app, a handoff-ready doc site (no manual cleanup), a test report, and a full build timeline.
WF-2 — gated build + gap loopA build that hits a failing gate (e.g. 3/50 failing) runs the gap loop and re-converges so the same locked criteria pass, with every fail/fix visible in dashboard, docs, and the decision point.
WF-3 — stakeholder docsAfter deploy, the stakeholder can open and navigate the documentation site — phases and test coverage included — with no manual cleanup required.
WF-4 — live monitoringA per-role terminal whose heartbeat exceeds the threshold is flagged for restart in /forge status and the observatory; live terminals keep building; persistent stalls escalate to the operator after 5 same-phase nudges.
Why this matters. Because "done" is defined by tests generated from the plan before any code is written, every workflow's expected data-out is locked as an assertion up front. The gap loop (WF-2) is what closes any distance between the current build and these assertions.

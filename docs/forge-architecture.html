FORGE Architecture Observatory Spec Comparison
FORGE
FORGE is a dynamic workflow template that generates a project-specific build system every time it runs. You give it a product idea, and it assembles the right team of AI agents, the right expert panels, the right safety checks, and the right test strategy for that specific project. No two FORGE builds produce the same architecture. The pipeline structure is fixed — 12 phases, 4 human checkpoints — but everything inside is composed dynamically from a library of thinking frameworks. It stops at four checkpoints for your approval, and each build makes the next one smarter through compound learning.
Technically: a 12-phase pipeline with 4 safety mechanisms, a 3-terminal architecture during build (orchestrator + supervisor + watchdog via tmux/claude-peers), tiered multi-agent code review (5 specialized reviewers), document review agents at every gate, compound learning capture, and a post-ship product pulse loop. Start with D0 (full architecture), then drill into D1–D20 for detail.

Core Principle
Separate the builder from the auditor — the single design decision that shapes every mechanism in FORGE
When the same agent implements code and audits it, the implementor always wins. It produces visible progress — a working app, screenshots, a deploy. The auditor produces invisible safety — tests, assertions, coverage checks. Under time pressure or context pressure, invisible work gets skipped. Every time. This isn’t a discipline problem. It’s an incentive misalignment that no amount of prose instruction can fix.
The skip pattern — why prose enforcement fails
Protocol says “do X before proceeding”
X doesn’t produce visible outputtests, assertions, reviews, coverage checks
Next step DOES produce visible outputworking code, screenshots, deploy, demo
Agent skips X and does the exciting thingevery time, under any pressure
The structural fix — four mechanisms that replace prose with enforcement

R1: Phase Gate Hook

A shell script that blocks STATE.json writes unless required artifacts exist. Mechanically impossible to advance phases without the proof. forge-phase-gate.sh checks: eval harness before build, P6_EXIT.json before test, WALKTHROUGH.md before gaps.

R2: Separate Watchdog Terminal

A persistent tmux terminal whose only job is auditing. No implementation tasks, no progress incentive. Runs drift checks on every PR and merge, plus periodic 30m audit. Cannot be pressured by the implementor’s context.

R4: Blocking Dependencies

Safety steps are tasks in the dependency graph, not prose reminders. The eval harness is task-00 — every other task depends on it. The build literally cannot start until tests exist. A skippable step becomes an impossible-to-skip dependency.

CE: Independent Reviewers

5 specialized reviewer subagents run in their own context, each with a single lens (correctness, security, spec, performance, maintainability). They cannot be influenced by the implementor’s momentum or the operator’s impatience. Confidence-gated verdict: any HIGH-confidence objection blocks the PR.
The test: if you see a step in this document described in words but not enforced by a hook, a dependency, or a separate agent — that step will eventually be skipped. Flag it.
Discovered through real build failures. The orchestrator accurately quoted protocol requirements when asked — it knew the rules. It skipped them anyway because the implementor role produces visible progress and the auditor role doesn’t. The fix is never “try harder.” The fix is always structural.

D1 — Master Pipeline
The complete pipeline: 12 phases, 4 gates, 1 feedback loop, 1 compound loop
The pipeline is the sequence of steps FORGE follows to turn an idea into working software. It has four acts: first, understand what needs to be built (decompose the PRD, consult expert panels, do research). Second, write a detailed plan. Third, build and test the software with a 3-terminal agent team, then fix anything that doesn’t work. Fourth, after shipping, monitor how the product is actually used and feed that signal back into the next build. The system stops four times for human approval — after PRD decomposition, after research, after architecture, and before shipping.

P10 = Compound (post-ship pulse feeds next build). The gap loop (P8) re-derives the spec on failure. The compound loop (P10 → P0) makes each build smarter than the last.
Eval-first / TDD invariant. Before P6 begins, P5 generates an immutable eval harness from the workflow state machine (P4). Tests are written before code, generated from spec — not by the implementing agent. Agents cannot modify their own success criteria. The branch only advances on verified improvement (keep-or-revert ratchet). Sources: Karpathy autoresearch loop (program.md: "prepare.py is read-only"); Kent Beck, Test-Driven Development (2002): red-green-refactor with the test surface owned by a separate concern.
Inter-stage assertions. Between each phase, programmatic contract checks run automatically. After P4: every PRD requirement maps to a spec section or explicit out-of-scope tag. After P5: every task references a spec section; dependency graph has no cycles. After P6: every file in the repo is owned by a spec section. Assertion failure = automatic retry with the violation message appended. Source: Zaharia et al., "The Shift from Models to Compound AI Systems" (Berkeley AI Research, 2024); DSPy Assert/Suggest pattern.
D2 — Agent Topology
10 agents mapped to pipeline phases, each selected for isolation and cost
FORGE uses 10 specialized AI agents, each with a specific job. One orchestrates the whole process. Others run expert panels, do research, write the spec, build the code, review it, test it, and watch for problems. They’re organized like a small company: there’s a manager (supervisor), a team of up to 4 builders working in parallel, a quality inspector (watchdog), and a tester. Each agent only sees and touches what it needs to — this prevents one agent’s mistake from cascading into everything else.
References D1 phases. Three primitives: skill (orchestrator's context), agent/subagent (own context), prompt injection (behavioral rule). See D7 for fan-out sizing.

D3 — Human Gates
4 decision points where the pipeline pauses for operator input — each pre-screened by document review agents
FORGE works autonomously, but it stops four times to ask you for approval. Before each stop, document review agents (feasibility, scope guardian, coherence, adversarial) pre-screen the artifacts so you see higher-quality material. The first stop is after decomposing the PRD into a checklist — “did I understand the requirements correctly?” The second is after expert panels and research — “here’s the direction, should I write the spec?” The third is after the full spec — “here’s the architecture, should I start building?” (point of no return). The fourth is after building and testing — “should I ship?”

D4 — Drift Detection
7-category watchdog: triggered on PR submission, merge, and periodic audit — never on work-in-progress
The watchdog is like a quality inspector who checks every piece of work before it gets merged into the final product. It looks for seven types of problems: Did the code match the plan? Did the interfaces stay consistent? Did anyone add features that weren’t in the spec? Did the tests get weakened? The key design choice is that the watchdog only checks work that’s claimed to be finished — it doesn’t interrupt builders mid-task, because unfinished work will always look broken.
References D1 P6. The watchdog does not run on a blind timer against in-progress worktrees — agents mid-implementation will always fail checks because the work isn't done. Instead, it triggers at 3 specific points where the code is claimed to be complete.

D5 — Decision Router
How agents classify decisions: act silently, log, escalate, or queue for gate
While building, agents constantly face decisions — which variable name to use, which library to pick, whether to change an interface. The decision router tells every agent how to handle each decision based on how hard it is to undo. Trivial choices (naming, formatting) are made silently. Medium choices (library picks) are made and logged. Big choices (adding a dependency, changing an API) get escalated to the supervisor. Irreversible choices (dropping a feature, changing scope) are held for a human gate. This keeps the build moving fast on the small stuff while protecting against costly mistakes.
Injected into every agent prompt (D2). Prevents agents from pausing the build for trivial decisions or making irreversible choices without approval.

D6 — Research Protocol
How panels and research agents produce grounded, traceable evidence
When expert panels raise questions (“which database should we use?” or “can this API handle our volume?”), a research agent investigates and comes back with at least three options and a recommendation. Every finding gets labeled: VERIFIED means it was confirmed with a real source or test. UNVERIFIED means it’s a reasonable assumption but hasn’t been proven. OPEN means the research timed out and a human needs to weigh in. This prevents the system from building on top of guesses.
References D1 P2–P3. Every claim classified as VERIFIED, UNVERIFIED, or OPEN. OPEN questions surface at G1 (D3).

D7 — Fan-Out Sizing
How the system determines implementor count during P5 (Task Decomposition)
FORGE can run up to four builders working at the same time, but it doesn’t always use all four. A small project with five simple tasks gets one builder. A medium project with a dozen tasks gets two. A complex project gets three or four. The cap at four exists because someone has to review all the work — and reviewing more than four parallel streams honestly is impossible. If a project needs more parallelism, FORGE runs two sequential rounds of four rather than fanning out to eight.
References D1 P5, D2 implementor row. Based on task count, dependency parallelism, and reviewer capacity. Cap: ≤4 (beyond 4, review quality degrades).

D7b — Test-Driven Development
Tests generated from workflow state machine before any code is written
Before any code is written, FORGE generates a set of tests that define what “working” looks like. These tests come from the user workflow map — each step in the user’s journey becomes a test case. The critical rule: the building agents are not allowed to change these tests. They can only write code that passes them. This prevents a common failure mode where an AI “fixes” a failing test by weakening the test instead of fixing the code.
References D1 P4–P5. The eval harness is immutable once generated — implementing agents cannot weaken, skip, or modify tests. This is the TDD red-green-refactor cycle, enforced structurally rather than by discipline. Sources: Kent Beck, TDD (2002); Karpathy autoresearch "prepare.py is read-only."

D8 — Build & Bug-Fix Loop
How code is written, reviewed, and fixed without human intervention
When an agent writes code, it immediately runs the tests. If the tests pass, the code gets submitted for review and merged. If they fail, the agent gets three tries to fix the problem. There’s a critical safety rule: the codebase can only get better, never worse. If a change makes previously-passing tests fail, the change is automatically undone. This “ratchet” guarantees that the project moves forward or stays put — it never regresses. After three failed fix attempts, the problem gets escalated rather than endlessly retried.
References D1 P6. Implements keep-or-revert ratchet: the branch only advances on verified improvement. Failed tests trigger a 3-attempt fix loop before escalation. Source: Karpathy autoresearch (program.md: "improvements advance the branch; regressions git reset --hard").

D9 — Gap Loop + Optimization
The convergence mechanism: classify failures, re-derive spec, rebuild — plus parallel optimization experiments for performance gaps
After building and testing, problems are inevitable. The gap loop is how FORGE turns failures into fixes systematically. Each problem gets classified: is this a spec issue (the plan was wrong) or a PRD issue (the requirements were ambiguous)? D15 — Trust Model

D19 — Dynamic Composition
The pipeline is a fixed scaffold — everything inside is composed dynamically per project from a library of thinking frameworks
FORGE doesn’t run the same way twice. Each project gets a custom-built system assembled from reusable frameworks. A healthcare security project gets security-weighted reviewers, a children’s ed-tech project gets accessibility-weighted reviewers. The pipeline order never changes, but the agents, panels, pillars, tests, and review criteria are all selected at runtime based on what the project needs. The orchestrator’s job is to pick the right frameworks for this specific build.
No two FORGE builds produce the same architecture. The pipeline (D1) is the deterministic scaffold. The content is composed from a framework library. When a project needs a capability that doesn’t map to an existing framework, the orchestrator logs it as a gap and uses the closest available framework with documented adaptations.

D20 — Compound Engineering Integration
Seven capabilities from Compound Engineering make each build smarter than the last — the system compounds, not just the code
Traditional builds accumulate technical debt. Each feature adds complexity. Each bug fix leaves behind local knowledge that someone has to rediscover later. The system gets smarter with use.

D21c: Expert panel skill pattern — the template for creating new panels
Every expert panel skill follows the same structure. The skill is stable — it doesn’t change between projects. The panelists are named, opinionated, and each has a fixed lens. The synthesis is the skill’s output, not any single panelist’s opinion. Here is the pattern, using the technical-expert-panel as the worked example.

Skill Structure (5 sections)

1. Frontmatter — name, description (with trigger phrases), version
2. Intro — “When invoked, you convene an N-person panel…” + anchor document paths + other load-bearing docs to read first
3. How the panel works — frame in one sentence, each panelist 1–3 bullets in their own voice, synthesis at the end
4. The panel — one subsection per panelist with fixed lens
5. Output format — markdown template, then rules + when not to run

Panelist Definition (3 fields)

Name + Title — e.g., “Olivia Park — Senior Frontend Engineer (React / TypeScript)”

Owns: what this panelist is the authority on. Defines their lane.

Reflexively pushes back on: their automatic red flags. What they object to before thinking.

Reference frame: one litmus-test question they always ask. e.g., “Could a second engineer add a new dialogue branch without touching component code?”

Panel Skill Rules (apply to all panels)

1. Read before you speak. If a panelist’s lens touches the PRD or a code file, read it first. Do not comment on unseen artifacts.
2. No diluted voices. If you’re writing the same bullet for two panelists, one of them passes.
3. Be specific. Cite a file path, a CSS token, a measured number. Generic bullets are worse than silence.
4. Partial panels are fine. For narrow questions, run 3–5 voices and say which were skipped.
5. No panelist is the boss. The synthesis is yours, not theirs. The panel surfaces tensions; you resolve them.
6. No emojis. Match the workspace voice.
This pattern applies to all panel skills: technical, business, security, ASL, user, and project-specific.

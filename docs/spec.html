Section 1
Architecture — the complete system from pipeline to data flow
FORGE is a dynamic workflow template that generates a project-specific build system every time it runs. The pipeline structure (12 phases, 4 gates) is the deterministic scaffold. Everything inside — which panels run, which reviewers are weighted highest, what the tests check, what the Constitution's overridable articles say — is composed dynamically from the input project and a library of thinking frameworks.

Governing Principle

Separate the builder from the auditor. The agent implementing the code must never be the same agent auditing it. When one agent does both, the implementor always wins — it produces visible progress, while the auditor produces invisible safety. Under pressure, invisible work gets skipped. This is an incentive misalignment that prose instructions cannot fix. Every enforcement mechanism in FORGE exists because of this principle.

The test: if a step is described in words but not enforced by a hook, a dependency, or a separate agent — that step will eventually be skipped.

Pipeline: 12 Phases, 4 Arcs, 4 Gates
Work flows through three modes: understanding requirements (Arc 1), specifying the solution (Arc 2), and executing the build (Arc 3). A fourth arc handles post-ship compound learning. The transition between arcs is gated by human approval.

3-Terminal Architecture During Build Phase (P6)
During P6, three persistent tmux terminals run simultaneously: the orchestrator monitors lifecycle, the supervisor assigns tasks and coordinates reviews, and the watchdog drift-checks independently. Implementors are spawned as short-lived worktree agents by the supervisor.

Orchestrator

Spawns supervisor + watchdog at P6a. Monitors liveness via list_peers every 5m. Receives DRIFT_CRITICAL (emergency halt) and P6_COMPLETE (proceed to shutdown). Manages the shutdown handshake at P6e.

Supervisor

Reads TASKS.json. Assigns tasks to worktree agents in dependency order. Runs smoke tests on every return. Spawns 3–9 reviewer subagents per PR. Coordinates with watchdog via PR_SUBMITTED / DRIFT_RESULT messages. Sends P6_COMPLETE when all tasks have APPROVE verdict.

Watchdog

Triggered on PR_SUBMITTED, PR_MERGED, and /loop 30m. Drift-checks 7 categories against spec.md and CONTRACTS.md. Sends CLEAN/DRIFT/CRITICAL to supervisor. On CRITICAL: sends DRIFT_CRITICAL directly to orchestrator (bypass path). Writes all results to AUDIT.json.

Safety Mechanisms: Four Structural Enforcement Patterns
Prose instructions fail under pressure. Each safety mechanism replaces a prose instruction with structural enforcement.

R1: Phase Gate Hook

forge-phase-gate.sh blocks STATE.json writes unless required artifacts exist. Phase 2 requires PRD-ENHANCED.md. Phase 6 requires EVAL/ with test files and TASKS.json. Phase 7 requires P6_EXIT.json with all assertions passing. Phase 8 requires WALKTHROUGH.md.

R2: Separate Watchdog Terminal

A persistent tmux terminal whose only job is auditing. No implementation tasks, no progress incentive. Spawned automatically at P6a — not discretionary. P6 exit assertions verify AUDIT.json has watchdog entries.

R4: Blocking Dependencies

The eval harness is task-00 in TASKS.json. Every other task depends on it. The build literally cannot start until tests exist. A skippable step becomes an impossible-to-skip dependency.

CE: Independent Reviewers

3–9 specialized reviewer subagents run in their own context, each with a single lens (correctness, spec, security, performance, maintainability, API contract, reliability, pattern recognition, standards). Any HIGH-confidence objection blocks the PR.

Inter-stage assertions (Zaharia et al., Compound AI Systems) run between phases: after P4, every PRD req maps to a spec section. After P5, every task references a spec section with no dependency cycles. After P6, every file is owned by a spec section. Assertion failure triggers automatic retry with the violation appended.

Category 1
Pipeline Structure
Requirements governing the sequencing of work from idea to deployed application. The current reference implementation is 12 phases (P0–P10) in 4 arcs.
PIP-01The pipeline must decompose work into sequential phases with explicit boundariesMUST
Work must flow through discrete phases, each with defined inputs, outputs, and completion criteria. No phase may consume output from a phase that has not completed. Acceptance: Every phase writes its output to a designated file before the next phase reads it. STATE.json records the current phase.
Ref: D1, SKILL.md PipelineStatus: ImplementedImpl: STATE.json + forge-phase-gate.sh (R1)
PIP-02The pipeline must organize into three arcs: Understand, Specify, ExecuteMUST
Work must proceed through three distinct modes: understanding requirements, specifying the solution, and executing the build. The transition between arcs must be gated by human approval. Acceptance: Human gates exist at each arc boundary.
Ref: D0, D1, deck.mdStatus: Implemented
PIP-03Phase advancement must be structurally blocked without proof artifactsMUST
The system must mechanically prevent advancing to the next phase unless the current phase's required output artifacts exist and are non-empty. Prose instructions are insufficient — enforcement must be via hooks. Acceptance: A shell hook blocks STATE.json writes unless required artifacts exist. Rationale: Meridian build showed the orchestrator skipped phases under context pressure despite knowing the rules.
Ref: R1 (Changelog v2.4)Status: ImplementedImpl: forge-phase-gate.sh

All pipeline state must be persisted to files, not conversation context. After a crash, context limit, or deliberate pause, the system must resume from the last completed phase by reading state files. Acceptance: /forge --resume reads STATE.json and continues from the incomplete phase.

Section 8
Agent System Evaluation — observability, traceability, and eval for every agent graph in the project
When a FORGE project builds an agentic system (multi-agent orchestration, LLM pipelines, tool-using agents), each agent graph gets its own eval directory with dedicated observability, traceability, token monitoring, and logging. This section defines the taxonomy, the dynamic generation prompt, and provides an illustrative template using AgentForge (adversarial AI security platform) as the example.

Why agent eval is separate from application eval

EVAL/ (existing, TDD-01) tests the built application — does the app work as specified? Tests derived from the workflow state machine.

EVAL-AGENTS/ (new) tests the FORGE build agents themselves — did the orchestrator, supervisor, watchdog, and implementors perform well? Token efficiency, task completion rate, drift frequency, stall patterns.

EVAL-{system-name}/ (new) tests agent systems the project builds — if the app contains agent graphs (like AgentForge’s Red Team, Judge, Orchestrator), each gets its own eval with its own observability. A project may have 0, 1, or many of these.

Agent Graph Runtime — native execution vs. LangGraph adapter
FORGE’s eval framework is runtime-agnostic: GRAPH.md is the spec, JSONL traces are the contract. The runtime that executes the agent graph is a project-level decision made at P4 (spec) and configured in forge.yaml. Two options are supported. Both produce the same trace format, so the eval dashboard, tests, and observability work identically regardless of choice.

Decision criteria

Choose native when: the agent graph is simple (≤5 nodes, no cycles), you want zero dependencies, agents are Claude Code subagents or direct API calls, or you need full control over execution.

Choose LangGraph when: the graph has cycles or conditional loops, you need built-in checkpointing for long-running graphs, you want LangSmith’s trace visualization and eval tooling, or the team already uses the LangChain ecosystem.

This is a COULD requirement — LangGraph is a later-phase enhancement. The native runtime ships first. LangGraph adapter is added when a project needs it.

# Auto-generated from .forge/EVAL-red-team/GRAPH.md at P6 # Runtime: langgraph (configured in forge.yaml) from langgraph.graph import StateGraph, END from langchain_anthropic import ChatAnthropic from typing import TypedDict, Annotated import operator class RedTeamState(TypedDict): target_system: str attack_categories: list[str] results: Annotated[list, operator.add] # Reducer: accumulate results def orchestrator(state: RedTeamState) -> dict: # Routes to specialist agents based on attack_categories ... def prompt_injection(state: RedTeamState) -> dict: # Attempts injection attacks, returns findings ... def data_exfil(state: RedTeamState) -> dict: # Attempts exfiltration, returns findings ... def jailbreak(state: RedTeamState) -> dict: # Attempts guardrail bypass, returns findings ... def route_attacks(state: RedTeamState) -> list[str]: # Conditional edge: dispatch to relevant specialists return [cat for cat in state["attack_categories"]] # Build graph from GRAPH.md spec graph = StateGraph(RedTeamState) graph.add_node("rt-orchestrator", orchestrator) graph.add_node("rt-prompt-injection", prompt_injection) graph.add_node("rt-data-exfil", data_exfil) graph.add_node("rt-jailbreak", jailbreak) graph.set_entry_point("rt-orchestrator") graph.add_conditional_edges("rt-orchestrator", route_attacks, { "injection": "rt-prompt-injection", "exfiltration": "rt-data-exfil", "jailbreak": "rt-jailbreak", }) # All specialists return to orchestrator for synthesis graph.add_edge("rt-prompt-injection", "rt-orchestrator") graph.add_edge("rt-data-exfil", "rt-orchestrator") graph.add_edge("rt-jailbreak", "rt-orchestrator") app = graph.compile(checkpointer=SqliteSaver.from_conn_string("red_team.db"))
Example: Trace adapter — LangSmith callback → FORGE JSONL
# Translates LangSmith run traces to FORGE eval trace format # Runs as a post-execution hook, not inline from langsmith import Client def langsmith_to_forge_traces(project_name: str, eval_dir: str): client = Client() runs = client.list_runs(project_name=project_name, is_root=True) for root_run in runs: trace_id = root_run.id child_runs = client.list_runs(parent_run_id=trace_id) lines = [] for seq, run in enumerate(child_runs, 1): lines.append({ "trace_id": str(trace_id), "seq": seq, "agent": run.name, # Maps to GRAPH.md agent name "input_tokens": run.prompt_tokens, "output_tokens": run.completion_tokens, "latency_ms": run.latency_ms, "model": run.extra.get("model", ""), "tools_called": [tc.name for tc in run.tool_calls], "decision": run.outputs.get("route", None), "output_summary": str(run.outputs)[:200], "error": str(run.error) if run.error else None, "parent_seq": None # Resolved from run.parent_run_id }) # Write FORGE-format JSONL outpath = f"{eval_dir}/TRACES/{root_run.start_time:%Y-%m-%dT%H-%M-%S}-{trace_id[:8]}.jsonl" with open(outpath, "w") as f: for line in lines: f.write(json.dumps(line) + "\n")
The key constraint: Both runtimes converge at TRACES/*.jsonl. The eval dashboard, test runner, token monitor, and metrics calculations all read JSONL — they never call LangSmith or LangGraph APIs directly. This means you can switch runtimes per agent graph without touching the eval infrastructure. A project could use native for simple graphs and LangGraph for complex ones simultaneously.

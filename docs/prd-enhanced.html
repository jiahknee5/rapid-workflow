Enhanced PRD · Requirements Register
A machine-traceable requirements register with change tracking
Derived from docs/PRD.md + skills/forge/SKILL.md. This is the requirements doc that 04-spec/spec.md and .forge/TASKS.json reference; the HTML deck is its developer-view presentation.

1 · Enforcement over prose

Every safety step is a hook, a dependency, or a separate agent.

2 · The writer is never the auditor

Implementation and review are structurally split.

3 · The gap loop is the product

The build is the artifact; surfacing and closing gaps is the value.

How to read this register. Requirements live at one of three altitudes — a pyramid: Business (BR) = why / outcomes → Functional (FR) = what the system does → Technical / NFR (TR) = how + constraints. Each lower requirement traces ↑ up to its parent and (for TR) ↓ down to spec / architecture. Priority (MUST / SHOULD / COULD) is a badge on each requirement, not the grouping. Every requirement carries a change history; all changes are consolidated in the Change Log, logged when requested (requested → accepted → implemented) so the PRD is an auditable diff, never silently edited.

Each entry records the source/issue that drove it and the requirement(s) it updated, so any GitHub issue traces to exactly where the PRD changed — and back. A change is logged when requested (status: requested → accepted → implemented).

Change-request protocol. When a requirement is requested to change: (1) file/identify the GitHub issue (gap loop), (2) add a row here with requested status naming the issue and the affected BR/FR/TR, (3) update the requirement + its per-requirement change history, (4) flip the row to implemented when shipped (and gaps-to-issues.sh closes the issue). This is the single place "where the PRD gets updated" is recorded.

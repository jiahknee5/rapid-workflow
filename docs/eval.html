Evaluation · Overview
How RAPID evaluates a build
RAPID's evaluation system has two jobs. First, it proves that the FORGE enforcement hooks themselves still work — that is the immutable task-00 smoke test. Second, at the end of the build it runs a ship gate that refuses to let the build advance to walkthrough while stub or conformance gaps remain open. Both are file-backed and re-runnable, so the result is always derived from the current tree rather than a remembered claim.

The two evaluation surfaces

task-00 harness (.forge/EVAL/hooks.smoke.test.sh) — an immutable smoke test that builds a throwaway .forge tree in a temp directory and asserts the hooks for R1, R7, R8 and R9 behave correctly.

Ship gate (.forge/P6_EXIT.json) — three exit assertions written at P6 that the phase gate (S-01) reads to decide whether P7 (walkthrough) may begin.

Evaluation · Harness
The eval harness — task-00, the immutable smoke test
.forge/EVAL/hooks.smoke.test.sh is the dependency root of the build (per spec section S-04: "task-00 eval-harness is the dependency root in .forge/TASKS.json"). It is re-runnable: it locates the repo's tools/ directory, makes a temporary build skeleton (.forge, 01-intake, 04-spec, src), seeds it with {"phase":1}, and exercises each enforcement hook against it. It exits non-zero if any assertion fails.
What it asserts
Four checks, each tied to a requirement. The harness uses an assert helper that compares an actual value to an expected one and sets a fail flag.

R1 — Phase gate

Writing STATE.json with phase:2 while PRD-ENHANCED.md is absent must be blocked (the hook exits 2). After the artifact is created, the same write must be allowed (exit 0).

R7 — Continuation hook

With the phase artifact removed, a Stop event must be blocked — the stop-hook output must contain "decision": "block".

R9 — Stub detection

A source file containing // TODO implement must cause the stub-detect hook to file a stub gap — a GAP-STUB entry must appear in .forge/GAPS.json.

R8 — Conformance plumbing

R8 (module conformance) is part of the same hook family this harness guards; the harness header names R1/R7/R8/R9 as the behaviors asserted through the shared tools/ hooks.
Key checks, verbatim from the harness
set -uo pipefail TOOLS="$(cd "$(dirname "$0")/../../tools" && pwd)" B="$(mktemp -d)"; trap 'rm -rf "$B"' EXIT cd "$B"; mkdir -p .forge 01-intake 04-spec src printf '{"phase":1}\n' > .forge/STATE.json fail=0 assert(){ [ "$1" = "$2" ] || { echo "FAIL: $3 (got '$1' want '$2')"; fail=1; }; } # R1: block phase advance without artifact, allow with it printf '{"tool_input":{"file_path":".forge/STATE.json","content":"{\\"phase\\":2}"}}' \ | bash "$TOOLS/phase-gate-hook.sh" >/dev/null 2>&1 assert "$?" "2" "R1 blocks P2 without PRD-ENHANCED" printf '# prd\n' > 01-intake/PRD-ENHANCED.md printf '{"tool_input":{"file_path":".forge/STATE.json","content":"{\\"phase\\":2}"}}' \ | bash "$TOOLS/phase-gate-hook.sh" >/dev/null 2>&1 assert "$?" "0" "R1 allows P2 with PRD-ENHANCED" # R7: block stop when phase artifact missing rm -f 01-intake/PRD-ENHANCED.md out=$(printf '{"hook_event_name":"Stop"}' | bash "$TOOLS/stop-hook.sh" 2>/dev/null) echo "$out" | grep -q '"decision": "block"'; assert "$?" "0" "R7 blocks premature stop" # R9: stub detected in source file printf '04-spec/spec.md\n# s\n## S1\n' > 04-spec/spec.md printf 'function f(){ // TODO implement\n}\n' > src/a.js printf '{"tool_input":{"file_path":"src/a.js"}}' \ | bash "$TOOLS/stub-detect-hook.sh" >/dev/null 2>&1 grep -q 'GAP-STUB' .forge/GAPS.json 2>/dev/null; assert "$?" "0" "R9 files stub gap" echo "task-00 smoke: $([ $fail -eq 0 ] && echo PASS || echo FAIL)" exit $fail
Source: .forge/EVAL/hooks.smoke.test.sh. The harness is "immutable" — it is the fixed reference the build is measured against, not a test that gets rewritten to pass. This page does not claim the harness currently passes; it describes what the harness asserts.
Evaluation · Ship gate
The ship gate — P6 exit assertions
At P6 the ship gate (tools/ship-gate.sh, spec S-09) merges the stub and conformance assertions into .forge/P6_EXIT.json. The phase-gate hook (S-01) reads that file and blocks P7 (walkthrough) if any assertion fails. The table below shows the assertions exactly as recorded in the current .forge/P6_EXIT.json.
StatusAssertionDetail (from file)
● PASS no_open_stub_gaps 0 open MAJOR+ stub gap(s)
● PASS no_stubs_in_tree 0 stub marker(s) found in source tree
● PASS no_open_conformance_gaps 0 open MAJOR+ conformance gap(s)
All three P6 exit assertions are recorded as passing in .forge/P6_EXIT.json (each entry has "pass": true). Because no assertion fails, the phase gate does not block advancement to P7. If any assertion's detail showed a non-zero count, its pass would be false, its dot would be red, and S-01 would refuse the walkthrough phase.
Source: .forge/P6_EXIT.json. Status dots reflect the file's recorded values; this page does not assert results beyond what the file contains.
Evaluation · Principle
Keep-or-revert principle
The eval harness is the fixed reference and is never weakened to make a build pass. A change is kept only if the relevant assertions still hold; otherwise it is reverted. The harness asserting exact exit codes (2 for blocked, 0 for allowed) and exact output strings ("decision": "block", the presence of a GAP-STUB entry) is what makes "keep" an objective, file-derived decision rather than a judgment call.

Why keep-or-revert is enforceable here

The smoke test is re-runnable against a throwaway tree, so it can be run before and after any change to the hooks or the build. If a change breaks an assertion, the harness exits non-zero and the change does not survive.

At ship time the same discipline applies through .forge/P6_EXIT.json: a build only ships forward when no open MAJOR+ stub or conformance gaps remain and no stub markers are found in the tree. A regression flips an assertion to "pass": false and the gate holds.

Evaluation · Gating
How tests gate phases (R1 and R4)
Two requirements connect the eval system to phase progression. R1 is the phase gate that blocks advancement without the prerequisite artifact; R4 makes the eval harness the dependency root so the build cannot reach P6 until tests exist.
R1 Phase-gate hook — block advance without the artifact Must
tools/phase-gate-hook.sh (spec S-01) runs PreToolUse on Write|Edit. It parses the written STATE.json content from stdin, reads the target phase, and blocks (exit 2) when the prerequisite artifact for that phase is absent. The artifact chain is: PRD-ENHANCED → synthesis → spec → EVAL+TASKS → P6_EXIT → WALKTHROUGH. At the P6→P7 boundary the prerequisite is a passing P6_EXIT.json.

R4 Blocking dependency graph — tests are the root Must
Spec S-04: the task-00 eval-harness is the dependency root in .forge/TASKS.json, and the phase gate refuses P6 until .forge/EVAL/ holds test files. This makes the smoke test a prerequisite of the build rather than an afterthought — there is no path to the ship gate that skips the eval harness.
Spec S-04 Root task task-00 Gate P6 requires .forge/EVAL/
The flow: R4 forces the eval harness to exist before P6 → the harness asserts R1/R7/R8/R9 still hold → the ship gate writes P6_EXIT.json → R1 reads that file and only then permits P7. Tests gate phases at both ends of the build.

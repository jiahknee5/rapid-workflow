Root — Cross-cutting
CLAUDE.md auto-loads for every conversation. CONSTITUTION.md, AGENTS.md, MEMORY.md alongside so every agent picks them up automatically.
00 — Vision · The Lens
One page company pillars, one paragraph project objective. Referenced by name in every panel prompt and every spec section.
01–03 — Inner Loop
PRD → Panels → Workflow. Iterate among these three until the workflow feels right. Then derive spec.
04 — Spec (incl. Swarm)
SPEC.md derived from PRD + workflow. Architecture, interfaces, and agents/*.md all live under 04-spec/ — the swarm topology is part of the spec.
05–06 — Build & Walkthrough
Swarm executes against the spec. Watchdog audits as CI hook. Agent walkthrough first; human is the final gap filter.
07 — Gaps · Outer Loop
Each gap classified spec-level (re-derive SPEC.md) or PRD-level (back to stakeholder). HISTORY.md is the audit trail of the loop itself.

Phase 1 · Intake
The PRD is a manual artifact — but the prompt pattern determines its quality
prompt pattern not a persistent skill — a one-shot structured prompt
Prompt to Build
Read this stakeholder brief. Structure it as
an immutable PRD with these sections:

- Problem statement (one paragraph)
- Target user (specific, not generic)
- Requirements (MUST / SHOULD / MAY)
- Constraints (the kind that eliminate
  architecture options — "from scratch",
  "runs offline", "iPad-first")
- Success criteria (measurable, not aspirational)
- Out of scope (explicit — saves weeks)

Freeze it. All future changes go through
spec diff in 04-spec/DIFF.md.
Constraints first — they eliminate 90% of options before you design
Frozen means frozen — changes tracked in DIFF.md, never overwritten
Same input every re-run — if the gap loop restarts, the PRD is identical
Invocation
> Here's the SuperBuilders brief. Our partner
> wants a browser-based ASL vocabulary app
> with from-scratch computer vision. Target:
> elementary students. Deadline: one week.
Example Output
## Constraints
MUST: From-scratch model (no pre-trained ASL)
MUST: Browser inference (<5MB model, no server)
MUST: Works on iPad Safari
SHOULD: Under 3s inference latency
MAY:  Support left-handed signers

## Out of Scope
- Sentence-level ASL (vocabulary only)
- Offline mode (requires camera + model)
- Android (iPad Safari only for v1)

These three constraints — from-scratch, browser,
iPad — eliminate 90% of architecture options.
The PRD surfaced them; the spec will honor them.
Why this works: The constraint "from scratch" shaped the entire architecture — model size, training pipeline, browser inference. Without surfacing it here, the spec would assume a pre-trained model and the build would fail at week's end.

Phase 2 · Review
Named panelists with fixed lenses — the build prompt defines their voice and boundaries
technical-expert-panel business-expert-panel user-panel
Prompt to Build
Create a Claude Code skill called
`technical-expert-panel`.

Convene 8 named experts, each with a fixed lens:
1. Sarah — Senior Frontend (React, state, renders)
2. Marcus — iPad/Mobile (touch, viewport, Safari)
3. Priya — Test Engineer (coverage, flaky, CI)
4. James — Accessibility (WCAG, screen readers)
5. Lin — Performance (bundle, LCP, memory)
6. Derek — DevEx/CI (builds, deploys, DX)
7. Elena — Security (XSS, CSRF, data exposure)
8. Alex — SWE Skeptic (over-engineering, yak shaving)

Rules:
- Each speaks 1–3 bullets, first person, own voice
- No filler. "Pass" if nothing useful to add
- Anchor to the project PRD — read before speaking
- End: convergent / divergent / recommended action
- Budget: 600–900 words
Named, not generic — "Marcus" has an opinion; "the mobile engineer" hedges
"Pass" is better than filler — forces honest signal, not noise
One skill per lens — business, technical, SME, user are separate skills
Invocation
/technical-expert-panel
Review the fraction manipulative component.
It uses client-side Canvas on iPad.
Example Output
Marcus (iPad): "Canvas touch events on iPad
Safari drop frames above 4 objects. Use CSS
transforms with will-change instead — I've
shipped this pattern on three production apps."

Lin (Perf): "The Canvas repaints on every drag.
That's 60fps of full-canvas clears. Measure actual
FPS on a real iPad Air, not just the simulator."

Alex (Skeptic): "Pass. Canvas is fine for 4
fraction pieces. Don't over-engineer this."

DIVERGENT: Marcus and Lin disagree on Canvas
vs CSS transforms for the manipulative.
RECOMMENDED: Prototype both approaches.
FPS test on a real iPad Air before committing.
Why this works: The skeptic said "pass" — that's signal. A generic review would have all 8 voices saying something, diluting the two voices (Marcus, Lin) that actually had the relevant expertise.

Phase 3 · Behavior
Grill-me forces every assumption into the open before you write a line of spec
grill-me interview relentlessly until shared understanding
Prompt to Build
Create a skill called `grill-me`.

Interview me relentlessly about every aspect
of this plan until we reach shared understanding.
Walk down each branch of the decision tree,
resolving dependencies one by one.

For each question:
- Provide your recommended answer
- Explain why
- Ask one at a time

If a question can be answered by exploring
the codebase, explore the codebase instead
of asking me.

Don't move to the next branch until the
current one is resolved.
One at a time — batched questions get shallow answers
Recommended answers — the skill proposes, not just interrogates
Codebase-first — don't ask what you can read
Invocation
/grill-me
Walk me through the user flow when
camera permission is denied.
Example Output
Q1: When the user denies camera access, what do
they see? A static fallback, or is the feature
completely unavailable?

Recommended: Static fallback with a "try again"
prompt. Blocking the entire feature for a single
permission denial loses 15–20% of first-time users.

Q2: If they grant camera later (via Settings),
does the app detect the change without a page reload?

Recommended: Use the Permissions API change event.
Let me check if Safari supports it...
[reads MDN docs]
Safari 16.4+. Your PRD says "latest Safari" — this
works. But define "latest" as a version number.

Q3: The fallback shows a static image. Which sign
does it show? The one the user was about to practice,
or a generic placeholder?
[exploring codebase: no fallback component exists yet]
Why this works: Q2 checked Safari docs instead of asking. Q3 read the codebase and found the fallback component didn't exist yet — surfacing a gap before spec, not after build.

Phase 4 · Design
Blueprint orchestrates a one-week build — three human gates, everything else autonomous
blueprint ticket in, deployed app out, one week
Prompt to Build
Create a skill called `blueprint`.

Ticket in. Deployed app out. One week.

The 11 primitives every factory needs:
 1. Record — BLUEPRINT.md (locked spec)
 2. Memory — append-only, one file per project
 3. Orchestrator — manual gates + /loop
 4. Execution — git worktrees (branch per task)
 5. Runtime — Claude Code CLI sessions
 6. Integration — MCP gateway + project MCPs
 7. Quality — gate-1, gate-2, gate-3 checklists
 8. Delivery — Vercel / Railway / custom
 9. Observability — /eval + session-logger
10. Skills — skill files for each agent role
11. Secrets — .env.example + verification

Three human gates. ~3 hours total human time.
The rest is autonomous.
Fixed appetite — Shape Up rule: cut scope, never extend time
Gates, not reviews — human says go/no-go, doesn't rewrite
Primitives checklist — verify all 11 exist before starting the pipeline
Invocation
/blueprint init
Project: ASL vocabulary practice app
PRD: docs/01-intake/PRD.md
Stack: Next.js 16, React 19, TailwindCSS v4
Deploy target: Vercel
Example Output
BLUEPRINT.md locked. Reading PRD...

Subtask decomposition (4 parallel agents):
  Agent 1: Camera + video pipeline
  Agent 2: Model inference + gesture matching
  Agent 3: Lesson UI + progress tracking
  Agent 4: Auth + database + API routes

Watchdog: monitors all 4 branches for spec drift.
CONTRACTS.md: interface boundaries defined.

File ownership (no overlap):
  Agent 1 owns: src/camera/, src/video/
  Agent 2 owns: src/model/, src/inference/
  Agent 3 owns: src/lesson/, src/components/
  Agent 4 owns: src/auth/, src/api/, prisma/

Awaiting GATE 1 approval before fan-out.
Why this works: File ownership eliminates merge conflicts. The watchdog can't fix code — only block merges. This separation keeps the review honest: builders build, reviewers review.

Phase 5 · Build
The swarm topology is spec'd in 04-spec/agents/ — each agent has a contract before it runs
agent contract orchestration pattern — each agent gets a brief
Prompt to Build
You are the orchestrator for this build.

Read SPEC.md and CONTRACTS.md. Decompose into
≤4 parallel implementation tasks. For each:

1. Create a git worktree branch
2. Write a task brief (what to build, interfaces)
3. Assign to an implementor agent
4. The watchdog audits every PR against:
   - Does it match the spec?
   - Does it break a contract?
   - Does it introduce scope creep?

Fan-out rules:
- Never more than 4 parallel implementors
- Each owns non-overlapping files
- Contracts define the interface, not the impl
- Watchdog can block merge, not fix code

Supervisor resolves conflicts between agents.
≤4 cap — more than 4 parallel agents and review quality collapses
Non-overlapping files — eliminates merge conflicts by design
Watchdog ≠ fixer — separation of powers keeps review honest
Dispatch
# Orchestrator dispatches after Gate 1 approval
Task 1 → worktree: feat/camera-pipeline
Task 2 → worktree: feat/model-inference
Task 3 → worktree: feat/lesson-ui
Task 4 → worktree: feat/auth-api
Watchdog Report
Watchdog — feat/model-inference PR #3:
  ✓ Matches SPEC §4.2 (inference pipeline)
  ✓ Respects contract: ModelResult interface
  ✗ Adds sharp dependency — not in approved stack
  BLOCKED: Remove sharp. Use canvas for image resize.

Supervisor — conflict resolution:
  Agent 2 and Agent 3 both import /lib/gestures.ts
  Contract says Agent 2 owns it.
  → Agent 3: consume the exported interface only.

Watchdog — feat/lesson-ui PR #5:
  ✓ Matches SPEC §5.1 (lesson flow)
  ✓ Touch targets ≥44px (Apple HIG)
  ✓ No scope creep detected
  APPROVED
Why this works: The watchdog caught an unapproved dependency (sharp) before it hit main. The supervisor resolved a file-ownership conflict using the contract, not opinion. Neither rewrote the agents' code.

Phase 6 · Validate
Click-by-click from named personas — not "users would probably..."
usability-testing UX self-audit from named user personas
Prompt to Build
Create a skill called `usability-testing`.

Run an honest UX self-audit from named user
personas, click-by-click against the live app.

Protocol:
1. Push back on terminology — "usability test"
   means human + observer. Offer instead:
   task scenarios, agent-driven sim, or both.

2. Identify 5–7 personas with distinct goals.
   Each has: name, role, goal, starting state,
   time budget.

3. Write each scenario click-by-click with the
   user's inner voice — what they expect vs.
   what actually happens at each step.

4. Optionally build a Playwright + Claude agent
   that executes scenarios automatically and
   catches regressions on every push.
Inner voice — "Is it broken?" catches what "user may be confused" misses
Terminology pushback — the skill corrects you before producing the wrong artifact
Agent runner — Playwright executes scenarios on every deploy, not just once
Invocation
/usability-testing
Run the kid and parent personas against the
deployed fraction lesson at tutor.vercel.app
Example Output
Persona: Eli, 9, iPad, first time

Step 1: Lands on /lesson. Sees "Let's explore!"
  ✓ Taps without reading — button is obvious.

Step 2: Fraction display shows 1/2.
  ✗ Eli tries to tap the fraction itself.
  Nothing happens.
  Inner voice: "Is it broken?"
  → ADD: tap feedback on fraction display

Step 3: Drag to split the bar.
  ✗ Drag handle is 24px. Eli's finger is ~40px.
  Misses on first try.
  Inner voice: "I can't grab it"

Phase 7 · Loop
The gap loop classifies and routes — spec-level re-derives, PRD-level escalates to stakeholder
loop-recipes loop schedule
Prompt to Build
Create a skill called `loop-recipes`.

Design patterns for /loop invocations. Instead of
one-shot tasks, give Claude a repeating job with:
- Check function (what to look for)
- Action function (what to do about it)
- Pacing (dynamic vs fixed cron)
- Stop condition

Core recipes:
1. Babysit a PR — fix CI, answer reviews, rebase
2. Monitor flaky tests — detect, re-run, report
3. Gap loop — classify each gap as:
   • Spec-level → re-derive SPEC from PRD
   • PRD-level → route to stakeholder
   Track in GAPS.md with severity + pillar.

The gap loop recipe is the most important.
It turns a one-shot build into a continuous
improvement cycle.
Classify, don't just log — spec gap and PRD gap have different owners
The loop is the product — the build is just the artifact it produces
HISTORY.md — the audit trail of the loop itself, not just the gaps
Invocation
/loop
Run the gap loop. Read agent-runs/ from the
walkthrough. Classify each finding.
Update GAPS.md.
Example Output
Gap #1: Camera fallback shows blank screen
  Classification: SPEC-level (missing error state)
  Pillar: Accessibility
  Severity: P1
  Action: Re-derive SPEC §3.4 — add fallback UI
  → Updated GAPS.md, routed to spec-author

Gap #2: Lesson assumes left-to-right reading
  Classification: PRD-level (locale missing)
  Pillar: Inclusivity
  Severity: P2
  Action: Route to stakeholder — is RTL in scope?
  → Added to GAPS.md, flagged for stakeholder

Gap #3: Touch target 24px on fraction handle
  Classification: SPEC-level (below Apple HIG)
  Pillar: Usability
  Severity: P1
  Action: Update SPEC §5.1 — min 44px targets
  → Spec re-derived. Build agent notified.
Why this works: Gap #2 didn't get fixed — it got routed to the stakeholder because "is RTL in scope?" is a PRD-level question. The classifier doesn't guess; it escalates what it can't resolve.

PHASE 1
PRD constraint "from scratch" shaped the entire build
The partner required from-scratch computer vision — no pre-trained ASL models. Surfacing this as a MUST constraint in Phase 1 eliminated 90% of architecture options before spec. Without it, the team would have built on a pre-trained model and discovered the requirement at demo day.
PHASE 2
Expert panels caught the MediaPipe violation before a single line shipped
Technical panel flagged that using MediaPipe at inference — a pre-trained Google model — violates PRD R7 (no pre-trained components). This became a constitutional article. The best-accuracy model (92.9%) was disqualified; the honest from-scratch model (74.2%) shipped instead. The panel saved the project from a false result.
PHASE 7
The gap loop drove model accuracy from 27.5% to 74.2% across 22 experiments
Each training run produced gaps classified by pillar and severity. Spec-level gaps (architecture fixes, data augmentation, rejection class) were re-derived automatically. PRD-level gaps (locale support, motion-sign scope) were routed to the partner. The gap loop wasn't a post-build retro — it was the 22-experiment training loop.
CROSS
10-article Constitution enforced at every merge — including against the best model
Article V (no pre-trained models) blocked the 92.9%-accuracy MediaPipe model. The framework forced the team to ship the honest result, not the impressive one. The Constitution didn't slow the build — it prevented a false demo that would have been caught by the partner.
The workflow's value isn't speed — it's honest signal. The gap loop found 16 issues. The Constitution blocked the best-looking model. The panels caught a requirement violation before build. Every phase added a filter that prevented a predictable failure from reaching the demo.

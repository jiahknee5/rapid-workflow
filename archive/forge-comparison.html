FORGE Architecture Observatory Spec Comparison

FORGE vs TDD Agent-Crew Scaffold

Both systems solve the same problem: how do you run multi-agent software builds without the AI agents cutting corners, skipping tests, or drifting from the spec? They arrive at surprisingly similar conclusions through different paths. This comparison maps where they converge, where they diverge, and what each can learn from the other.

Exhibit 2
Architecture comparison — both use role separation, but at different granularity

TDD Agent-Crew Scaffold

  • 3 roles: Team Lead (thin, durable) + Orchestrator (plans, reviews) + Implementer (per code area)
  • Communication via files + bounded direct messages
  • Team Lead persists across many orchestrator/implementer cycles
  • 4 escalation categories reach human via lead
  • Single-operator fallback for smaller projects
  • 13 slash commands
  • Project-agnostic — generates from your ARCHITECTURE.md

FORGE

  • 10+ roles: Orchestrator + Supervisor + Watchdog + Impl ×4 + Reviewer ×5 + Tester + Panelists + Researchers + Doc Review Agents
  • Communication via claude-peers MCP (12 message types, JSON envelope)
  • 3 persistent tmux terminals during build (P6)
  • 4 human gates with document review pre-screening
  • Fast track (1 panel, 1 impl) and full track
  • 12-phase pipeline with dynamic composition per project
  • Project-specific — generates pillars, panels, reviewers, tests from PRD

Exhibit 4
What each system should learn from the other

What FORGE should learn from the Scaffold

  • Step-9 hot routing is elegant. Captures lessons, doc notes, and future work during coding — not after. FORGE's Decision Router classifies decisions but doesn't route session-level learnings in real time.
  • Close-out gating is user-triggered. The scaffold's /session-end and /orchestrate-end run when the human decides, not at natural boundaries. FORGE's gates are at fixed pipeline positions — adding a "close out this session" command would help with context management.
  • Track-prefix naming for parallel work. The scaffold uses track prefixes (e.g., BE-, FE-) to prevent channel-bleed in multi-team work. FORGE's branch naming (forge/phase-6/{task-slug}) doesn't encode the track.
  • Bounded messaging budgets. The scaffold explicitly caps inter-agent message length. FORGE's claude-peers messages are unbounded — adding a budget would prevent context inflation.

What the Scaffold should learn from FORGE

  • Expert panels before coding. FORGE's P2 (3 panels: business, technical, SME) catches design flaws before implementation starts. The scaffold goes straight from architecture doc to tasks.
  • Phase gate hooks are structural. FORGE's R1 hook literally blocks advancement — the scaffold relies on user-triggered close-out, which can be skipped under pressure.
  • Tiered multi-agent review. 5 specialized reviewers with confidence gating catches more than a single orchestrator review. Different lenses find different bugs.
  • Compound learning across builds. FORGE's P10 (product pulse) → P0 (next build) loop means each build starts smarter. The scaffold's LESSONS.md captures per-project learning but doesn't feed it forward.
  • Visual QA and live observability. Playwright screenshots + Claude vision + a real-time Gantt dashboard are capabilities the scaffold doesn't address.
  • Document review agents at gates. Pre-screening gate artifacts with feasibility/scope/adversarial reviewers before the human sees them is a force multiplier.
Exhibit 5
The fundamental difference is enforcement philosophy — the Scaffold trusts process, FORGE trusts hooks
The TDD Agent-Crew Scaffold is a process-oriented system. It gives agents clear steps to follow (the 10-step /tdd walker), clear escalation rules (4 categories), and clear close-out procedures. It trusts that following the process produces good results. When the process fails, it adds more process.
FORGE started the same way — and learned the hard way that prose instructions fail under context pressure. An AI agent that is both the executor and the quality checker will cut corners when running low on context. FORGE's v2.4.0 (the R1–R6 fixes) moved enforcement from prose to structural mechanisms: hooks that block advancement, dependency graphs that make safety steps blocking, separate agents that audit independently, and concrete commands instead of vague directives.
The Scaffold is where FORGE was at v1.0. It has the right ideas — role separation, TDD enforcement, cross-doc invariants, lesson capture — but the enforcement model relies on discipline. FORGE's evolution shows that discipline fails under pressure, and the fix is mechanical enforcement: hooks, dependency graphs, and independent watchdogs. The Scaffold's strongest unique contribution is hot-routed Step-9 categorization — capturing learnings during coding, not after — which FORGE should adopt.
